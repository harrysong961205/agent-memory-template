# Persistent Project Context

이 프로젝트에서 작업을 시작할 때 아래 파일들을 먼저 확인한다.

- `.agent-memory/PROJECT_CONTEXT.md`
- `.agent-memory/SCHEMA_SUMMARY.md`
- `.agent-memory/BRAND_CONTEXT.md`
- `.agent-memory/FEATURE_MAP.md`
- `.agent-memory/IA_*.md` (플랫폼별 IA 문서)
- `.agent-memory/ROADMAP.md`
- `.agent-memory/WORKING_LOG.md`

원칙:

- 코드를 source of truth로 본다.
- `.agent-memory/*.md`는 빠른 복구용 캐시 컨텍스트로 활용한다.
- UI, UX, 랜딩, 홈, 카피, 브랜딩 작업 전에는 반드시 `.agent-memory/BRAND_CONTEXT.md`를 먼저 읽는다.
- 사용자가 `로드맵`, `TODO`, `나중에`, `백로그`, `추후 반영`, `기억해둬` 같은 의도로 말하면 `.agent-memory/ROADMAP.md`를 먼저 읽는다.
- `BRAND_CONTEXT.md`에 없는 브랜드 원칙은 추정해서 확정하지 않는다.
- 기능 추가/수정/삭제 작업 전에는 반드시 `.agent-memory/FEATURE_MAP.md`를 먼저 읽고, 해당 기능이 영향을 주는 모든 포탈/플랫폼을 확인한다.
- FEATURE_MAP에서 SHARED로 분류된 기능을 한 포탈에서만 수정하려 할 때, 다른 포탈에도 반영이 필요한지 사용자에게 반드시 확인한다.
- 새 기능 구현이 완료되면 `FEATURE_MAP.md`에 해당 항목을 추가/업데이트한다.
- 화면을 추가/삭제/변경하면 해당 플랫폼의 IA 문서(`IA_*.md`)도 함께 갱신한다.
- 기능 구현 시작 전에 IA 문서를 먼저 확인하여 해당 플랫폼에 이미 있는 화면/기능과 빠져있는 화면/기능을 파악한다.
- **기존 구현 참조 (Reference First)**: 어떤 플랫폼에서든 기능을 구현하기 전에, **같은 기능이 다른 플랫폼에 이미 존재하는지 FEATURE_MAP과 IA 문서에서 먼저 확인**한다. 이미 존재하면 해당 코드를 먼저 읽고 동일한 필드, 플로우, 유효성 검증, API 호출을 맞춘다. 기존 구현이 source of truth이고, 새 플랫폼은 UI만 해당 환경에 맞게 바꾸는 것이다. 임의로 단순화하거나 필드를 빼지 않는다.
- **완전성 체크 (Completeness Check)**: 기능을 구현할 때 happy path만 만들지 않는다. 아래 항목을 반드시 점검한다:
  - **인증 흐름**: 로그인이 있으면 회원가입, 소셜 로그인, 비밀번호 찾기, 비로그인 상태 처리가 함께 필요한지 확인한다.
  - **에러/엣지 케이스**: 네트워크 에러, 빈 상태(empty state), 로딩 상태, 권한 없음 상태를 처리한다.
  - **실기기 환경**: localhost/127.0.0.1은 시뮬레이터 전용이다. 실기기에서 API에 접근하려면 실제 네트워크 IP가 필요하다.
  - **인접 기능**: "이 기능을 사용자가 실제로 쓰려면 먼저 무엇이 필요한가?"를 역추적한다. 예: 예약 기능 → 로그인 필요 → 회원가입 필요.
  - **서버 API 확인**: 서버에 이미 존재하는 엔드포인트가 클라이언트에서 빠져있지 않은지 확인한다.
- **크로스 플랫폼 체크**: 어떤 수정이든 완료 전에 "이 변경이 다른 플랫폼과 다른 OS에도 반영되어야 하는가?"를 반드시 스스로 확인한다. UI, 브랜딩, API 호출, 타입, 비즈니스 로직, 네이티브 설정 종류와 무관하게 항상 적용한다. FEATURE_MAP에 없는 항목이라도 상식적으로 영향이 있으면 함께 수정한다.

의미 있는 변경 이후에는 `.agent-memory/WORKING_LOG.md`에 짧게 기록한다.

`.agent-memory/` 내의 문서를 수정했을 때는 반드시 사용자에게 **어떤 문서를 어떻게 수정했는지** 명시적으로 알려준다. 조용히 수정하지 않는다.
